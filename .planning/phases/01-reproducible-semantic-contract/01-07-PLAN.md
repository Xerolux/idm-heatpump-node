<interfaces>
`scripts/check-parity.mjs` supports explicit generate/check modes. With an optional explicit checkout argument it admits only a clean exact full-tag checkout. Without one, it clones/fetches only the allowlisted manifest repository into an owned temporary directory, checks out the manifest SHA with its exact tag available, verifies identity before import, prepares the fixed Python 3.12 reference environment, and cleans everything in `finally`.

The stable commands are `npm run parity:generate`, `npm run parity:api`, and `npm run parity:check`. They require no `IDM_HEATPUMP_API_DIR`; an explicit checkout option is an offline optimization, never an implicit prerequisite. All subprocesses use literal argument arrays with no shell.
</interfaces>

| -------------------------------------- | -------------------------------------------------------------------------------- |
| Manifest → temporary upstream checkout | Repository/tag/SHA/package/schema must be proven before any upstream import.     |
| Orchestrator → child processes         | Values and paths must remain literal fixed arguments and never enter a shell.    |
| Generator → repository outputs         | Generation must be atomic; check mode must never mutate authoritative artifacts. |

## STRIDE Threat Register

| Threat ID | Category               | Component            | Disposition | Mitigation Plan                                                                                   |
| --------- | ---------------------- | -------------------- | ----------- | ------------------------------------------------------------------------------------------------- |
| T-1-01    | Spoofing               | upstream identity    | mitigate    | Allowlisted URL, exact full SHA/tag/version/schema, full-tag checkout, verify before import.      |
| T-1-02    | Tampering              | generated outputs    | mitigate    | Pinned source, staging/atomic replace, non-mutating hard check, clean-diff assertions.            |
| T-1-03    | Repudiation            | orchestration result | mitigate    | Full provenance and bounded named-step diagnostics.                                               |
| T-1-04    | Information disclosure | process/logs         | mitigate    | No secrets/device inputs; synthetic facts only and bounded output.                                |
| T-1-05    | Denial of service      | clone/process/output | mitigate    | Fixed repository/tools/paths, bounded diagnostics, deterministic cleanup, no arbitrary recursion. |
| T-1-06    | Elevation of privilege | shell/path/import    | mitigate    | No shell, fixed executables, canonical owned paths, verification before import, cleanup.          |
| T-1-SC    | Tampering              | package supply chain | mitigate    | No npm expansion; exact audited Python reference dependency in isolated tooling only.             |
