<interfaces>
The only reserved exceptional-number envelope is `{ "$number": "NaN" | "+Infinity" | "-Infinity" | "-0" }`; sibling keys and unknown tags are invalid. Ordinary finite numbers and positive zero remain ordinary JSON numbers.

Each scenario requires `name`, `configuration`, `transport_responses`, `clock`, `operation`, `expected_result`, `expected_requests`, and `expected_state`, plus root schema and exact baseline identity. Raw words represented inside the contract transport/request envelope are integers in `0..65535`. This schema rule does not pre-validate direct codec operation values: codec vectors retain the datatype-specific accepted/rejected domains generated in Plan 02.
</interfaces>

<threat_model>

## Trust Boundaries

| Boundary                            | Description                                                                                      |
| ----------------------------------- | ------------------------------------------------------------------------------------------------ |
| JSON fixture → runtime parser       | Checked-in content still requires strict provenance, shape, operation, size, and tag validation. |
| Tagged JSON → numeric expectations  | NaN, infinities, and negative zero can be lost or confused with ordinary user strings.           |
| Contract words → later request data | Malformed transport/request words must fail without narrowing direct codec-operation semantics.  |

## STRIDE Threat Register

| Threat ID | Category               | Component              | Disposition | Mitigation Plan                                                                                    |
| --------- | ---------------------- | ---------------------- | ----------- | -------------------------------------------------------------------------------------------------- |
| T-1-01    | Spoofing               | fixture baseline       | mitigate    | Require full exact baseline identity and supported schema before accepting scenarios.              |
| T-1-02    | Tampering              | tagged/scenario values | mitigate    | Closed schemas, strict own-property parsing, deep cloning/freezing, and generated fixtures.        |
| T-1-03    | Repudiation            | validation outcomes    | mitigate    | Stable semantic codes and named mutation tests identify each rejected field/invariant.             |
| T-1-04    | Information disclosure | fixture content/logs   | mitigate    | Synthetic semantic values only; no credentials, PINs, IPs, identifiers, or raw captures.           |
| T-1-05    | Denial of service      | recursive structures   | mitigate    | Explicit depth/count/string limits, fixed fields/operations, and duplicate rejection.              |
| T-1-06    | Elevation of privilege | parser/object keys     | mitigate    | Plain own-property objects only; no reviver, eval, prototype merge, dynamic I/O, or child process. |
| T-1-07    | Tampering              | exceptional numbers    | mitigate    | One-key reserved tags and `Object.is`/`Number.isNaN` comparisons prevent lossy coercion.           |
| T-1-SC    | Tampering              | package supply chain   | accept      | No package/dependency changes; native runtime primitives and locked tests only.                    |

</threat_model>
